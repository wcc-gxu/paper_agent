## 1. 概览与重构原则

### 1.1 重构边界

| 维度 | 保留 | 改造 | 新增 |
|------|:---:|:---:|:---:|
| **Main Agent** | 6 节点 StateGraph 主体逻辑、安全双闸、evaluate 5 出口、outbox 消息机制 | 前端适配（iOS → Vue）、`ScenarioPlanResult` 扩展引用标记字段 | 冷启动引导流程、多用户 session 管理 |
| **子 Agent (Graphs)** | ingest_graph（保留核心编排）、rad_query_graph、clustering_graph、citation_chase_graph、translation_graph | history_graph 合并入 main_graph、video_graph 拆分重构 | — |
| **存储层** | PostgreSQL + pgvector + Redis、Checkpointer + Store 双系统 | AgentDB 增加 user_id 隔离字段、增加术语词表 collection | external_validations 缓存表正式启用 |
| **API 层** | FastAPI + WebSocket 主体、outbox_poller、auth 认证 | routes 增加多用户端点 | Vue 前端静态资源服务 |
| **Providers** | 7 个搜索源全保留 | — | — |
| **反幻觉** | 三层体系：人格设定 + 上下文质量 + 规则验证（见 anti-hallucination.md v2.0）| | 接入主链路、扩展为内外文献双通道验证 | groundedness + hallucination review |

### 1.2 三条不可妥协原则

1. **Main Agent 代码尽量不动** — 当前 6 节点 StateGraph 设计经过充分迭代，质量高，仅做必要扩展
2. **所有 LLM 输出可溯源** — 每一句话要么指向库内文献（`[local:paper_id]`），要么指向外部文献（`[ext:doi]`），要么显式标注 `[Agent 综合]`
3. **私有部署不做减法** — PostgreSQL + pgvector + Redis 全本地栈，不引入云服务依赖

## 2. 竞品分析结论摘要

> 详细报告见 [竞品对比分析报告](#附录-a-竞品对比分析报告)

### 2.1 关键发现

| 发现 | 对重构的指导 |
|------|-------------|
| **Scite 是唯一做"引用分类验证"的产品**（supporting/mentioning/disputing），其 Smart Citations 索引含 12 亿+条已分类引用语句 | 我们的 L2 CitationVerifier 不应止于"论文存在"，应增加"是否真的说了 LLM 声称的内容"（claim-level 校验） |
| **Elicit / Consensus 通过"不生成引用"来规避幻觉**（输出是数据库查询结果而非 LLM 自由文本） | 我们的路线不同——我们让 LLM 生成综合内容后再校验。这意味着校验必须更强 |
| **ReadPaper 是唯一有中文全文翻译的竞品**，但引用验证弱 | 中文市场是明确蓝海，跨语言检索 + 术语库是核心差异化优势 |
| **无一竞品做到端到端自动化**（搜索→下载→解析→入库→综述 全自动） | 保留这一定位，ingest_graph 的 7 阶段流水线是核心竞争力 |
| **无一竞品支持私有化部署** | 私有化部署是我们面向实验室场景的独特卖点 |

### 2.2 引用标记策略：竞品做法

| 竞品 | 做法 | 借鉴点 |
|------|------|--------|
| **Scite** | 每条引用语句可追溯到原文具体句子，标注 supporting/mentioning/disputing | 我们可输出 `[local:pap-xxx#sec-3.2]` 带章节定位 |
| **Elicit** | 输出结构化表格，每行对应一篇真实论文 | 综述输出可附带结构化 evidence table |
| **Consensus** | "Consensus Meter" 展示支持/反对分布 | 方法对比场景可展示不同论文的观点分布 |
| **Semantic Scholar** | TLDR 一句话摘要 + Highly Influential Citations 标注 | 论文入库后可自动生成 TLDR |

### 2.3 术语管理：竞品做法

- **ReadPaper**：内置中英术语对照，翻译时自动替换
- **Scite / Elicit / Consensus**：无术语管理功能
- **结论**：术语词表是我们可做的差异化功能，竞品无人涉足

## 3. Agent 架构重构：整合与拆分

### 3.1 现状分析

当前产品文档定义 6 个 Agent（Main / Literature / Knowledge / Research / Memory / Writing），重构新增 Glossary Sub-Agent，共 **7 个 Agent**（Main + Literature + Knowledge + Research + Writing + Capture + Glossary），代码层实际实现为：

```
产品文档 Agent        代码层 Graph             职责
─────────────────    ──────────────────      ──────────────────────
Main Agent           main_graph.py           路由、意图、编排
Literature Agent  ┐
Knowledge Agent   ├─ ingest_graph.py         搜索→评估→下载→转换→索引→排名→综述
Research Agent    ┐
                  ├─ clustering_graph.py     研究方向聚类
                  ├─ citation_chase_graph.py 引用追溯
Memory Agent      ┐
                  ├─ video_graph.py          视频解析（yt-dlp + ASR）
                  └─ history_graph.py        历史消息处理
Writing Agent     ── translation_graph.py    术语翻译
                  ── wiki_generator.py       Wiki 生成（非 graph）
                  ── rad_query_graph.py      RAG 问答
```

### 3.2 问题诊断

| # | 问题 | 影响 |
|---|------|------|
| 1 | `ingest_graph.py` 把 Literature（搜索/下载）和 Knowledge（解析/入库）混在一个 7 阶段 graph 里，违反产品文档的 Agent 边界划分 | 两个 Agent 的职责边界模糊，无法独立演进和复用 |
| 2 | Writing Agent 缺少核心能力——当前仅有 translation_graph，没有 citation 格式化、模板写作、术语词表管理等功能 | 与产品文档 §3.6 / §4 描述严重不匹配 |
| 3 | Memory Agent 名字与 LangGraph Store 的"长期记忆"概念冲突，且职责分散在 video_graph + history_graph 中 | video_graph 是采集工具，history_graph 是消息归档——两者都不叫"记忆" |
| 4 | `history_graph.py` 本质是 Main Agent 的消息归档，不应作为独立子 Agent | 应并入 Store 的 episodes namespace |
| 5 | `rad_query_graph.py` 和 `knowledge.py` 功能重叠 | 应统一为 Knowledge Agent 的 RAG 问答能力 |

### 3.3 重构后 Agent 架构

严格对齐产品文档的 6-Agent 划分，加上新增的 Glossary Sub-Agent，共 7 个：

```
┌─────────────────────────────────────────────────────────────┐
│                      Main Agent（保留不动）                    │
│   safety → intent → clarify → plan → execute → evaluate     │
│   6 节点 StateGraph + 5 出口 evaluate                         │

└──────────┬──────────┬──────────┬──────────┬─────────────────┘
           │          │          │          │
           ▼          ▼          ▼          ▼
    ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐
    │Literature│ │Knowledge │ │Research  │ │Writing   │
    │Agent     │ │Agent     │ │Agent     │ │Agent     │
    │          │ │          │ │          │ │          │
    │ 搜索     │ │ PDF解析  │ │ 聚类     │ │ RAG 问答 │
    │ 下载     │ │ 切片     │ │ 方法对比 │ │ 综述生成 │
    │ 元数据   │ │ embedding│ │ 趋势分析 │ │ 模板写作 │
    │ 筛选     │ │ 去重     │ │ 空白发现 │ │ 术语词表 │
    │          │ │ 增量更新 │ │ 引用追溯 │ │ 引用检查 │
    └──────────┘ └──────────┘ └──────────┘ └──────────┘
           │          │          │          │
           ▼          ▼          ▼          ▼
    ┌──────────────────────────────────────────────────────────┐
    │              Capture Agent（原 Memory Agent 改名）         │
    │  碎片知识采集：网页收藏 / 实验记录 / 会议笔记 / 音视频     │
    │  工具：video_graph（核心工具，非独立 Agent）               │

```

### 3.4 为什么 Memory Agent 改名 Capture Agent

| 原名称 | 问题 | 新名称 | 理由 |
|--------|------|--------|------|
| Memory Agent | 与 LangGraph Store 的"长期记忆"（preferences/profile/episodes）概念冲突 | Capture Agent | "采集"更准确描述其职责——把外部碎片内容（网页/视频/笔记/音频）采集到知识库 |

Capture Agent 作为独立 Agent 存在，但内部工具（video_graph / 网页收藏 / 笔记录入）可按需由 Main Agent 调度。

### 3.5 具体变更清单

#### 3.5.1 拆分：ingest_graph → Literature Agent + Knowledge Agent

当前 `ingest_graph.py`（7 阶段）需要拆为两个独立 graph。

**Literature Agent — `graphs/literature_graph.py`**（新建，从 ingest_graph 拆分）

| 节点 | 职责 | 来源 |
|------|------|------|
| `search` | 多源并发搜索（arXiv/S2/CNKI 等），支持术语词表 Query Expansion | ingest_graph search 节点 |
| `evaluate` | LLM 相关性评估 + 筛选 | ingest_graph evaluate 节点 |
| `download` | PDF 下载（HTTP + Playwright） | ingest_graph download 节点 |
| `convert` | **PDF → Markdown**（见下文 §3.5.1a） | ingest_graph convert 节点 |
| `extract_metadata` | 提取标题/作者/年份/DOI/摘要/期刊 | ingest_graph + provider 元数据 |

**Knowledge Agent — `graphs/knowledge_graph.py`**（新建，从 ingest_graph 拆分）

| 节点 | 职责 | 来源 |
|------|------|------|
| `chunk` | **双模式智能切片**（见下文 §3.5.1b） | ingest_graph index 节点前半 |
| `embed` | embedding 生成 + pgvector 入库 | ingest_graph index 节点后半 |
| `dedup` | 两级去重（精确 + 语义，见 §6） | 新增 |
| `rag_query` | RAG 问答入口 | 整合 rad_query_graph + knowledge.py |

##### §3.5.1a Literature Agent：PDF→MD 转换（LaTeX + 图片处理）

Literature Agent 的 convert 节点负责 PDF→MD 转换，需处理两个特殊问题。

**LaTeX 公式解析**：

```
PDF 中的公式通常是渲染后的图像或矢量图形，不是文本。

工具选型对比：
┌──────────────┬──────────────────┬──────────┬──────────┐
│ 工具          │ 公式处理方式     │ 质量     │ 速度     │
├──────────────┼──────────────────┼──────────┼──────────┤
│ pymupdf4llm  │ 保留为 $...$ 标记│ ★★★★    │ ★★★★★   │
│              │ (行内) $$...$$    │          │          │
│              │ (块级)，但不保   │          │          │
│              │ 证公式内容正确   │          │          │
├──────────────┼──────────────────┼──────────┼──────────┤
│ nougat       │ 专门为 LaTeX 设计│ ★★★★★   │ ★★      │
│ (Meta)       │ 输出完整 .tex    │          │ (需GPU)  │
├──────────────┼──────────────────┼──────────┼──────────┤
│ MathPix      │ OCR + LaTeX      │ ★★★★★   │ ★★★★    │
│              │ 渲染公式         │          │ (付费API)│
├──────────────┼──────────────────┼──────────┼──────────┤
│ Grobid       │ 结构化提取       │ ★★★     │ ★★★     │
│              │ (TEI XML)        │          │          │
└──────────────┴──────────────────┴──────────┴──────────┘

推荐：pymupdf4llm（默认）+ nougat（公式密集型论文的可选增强）

策略：
  - 默认使用 pymupdf4llm，速度快、质量可接受
  - 如果论文摘要中检测到高密度数学词汇（公式、证明、定理等关键词频率 > 阈值）
    → 标记为"公式密集型" → 改用 nougat 处理
  - 用户可在前端手动选择使用 MathPix（需用户提供 API Key）
```

**图片解析**：

```
PDF 中的图片（模型架构图、实验结果图、流程图）在 MD 转换中：
  - pymupdf4llm：图片导出为独立文件 + MD 中保留 ![img](path)
  - 图片 alt text 使用 PDF 中的 figure caption

进一步处理：
  1. 图片不直接参与文本分词（避免 caption 噪音）
  2. Caption 单独提取为特殊的 "figure_chunk"，关联到所属 section
  3. 可选：调用多模态 LLM（如 GPT-4V 或本地 VLM）对关键图片做描述
     → 对于术语提取，图片描述通常不是高价值信息，默认关闭

存图策略：
  ~/.paper_search/users/{user_id}/papers/{paper_id}/
  ├── paper.md           # 正文 Markdown
  ├── figures/
  │   ├── fig_001.png    # 架构图
  │   ├── fig_002.png    # 实验结果图
  │   └── fig_003.png    # 算法流程图
  └── meta.json          # 元数据
```

##### §3.5.1b Knowledge Agent：双模式智能切片

Knowledge Agent 的 chunk 节点需要支持两种切片策略：

```
模式 A：语义切片（RAG 检索用）
  用途：RAG 检索 / 论文精读 / 综述生成
  策略：Section-aware chunk（已有 chunker.py）
  大小：800-1500 tokens / chunk
  重叠：200 tokens
  产出 → pgvector table "paper_chunks"

模式 B：词粒度切片（术语提取用）
  用途：术语词表生成（Glossary Sub-Agent 调用）
  策略：按自然段切分 + 保留英文纯文本
  大小：不限制（按段落）
  后处理：
    - 去除 Markdown 标记（## /**/[]/()标记）
    - 保留 LaTeX 公式中的自然语言部分（$...$ 内容）
    - 按空格+标点做英文分词
    - 输入 CountVectorizer + TfidfVectorizer
  产出 → 内存中的词频统计 → Glossary Sub-Agent 流水线

触发：
  - Knowledge Agent 默认使用模式 A
  - Glossary Sub-Agent 触发 collect 时 → Knowledge Agent 追加执行模式 B
```

**两 Agent 的流水线编排**：Main Agent 通过 `depends_on` 串联：

```python
tools = [
    ToolCallSpec(call_id="0", kind="sub_agent", name="LiteratureAgent", arguments={...}),
    ToolCallSpec(call_id="1", kind="sub_agent", name="KnowledgeAgent",
                 arguments={...}, depends_on=["0"]),
]

**实现**：

| 组件 | 技术 | 说明 |
|------|------|------|
| Section 解析 | 正则提取 MD heading (## / ###) | 从论文 MD 中提取章节结构 |
| 结构聚类 | embedding + 相似度 | 对 Related Work 章节的段落主题聚类 |
| 模板库 | PostgreSQL 表 `writing_templates` | 预置 CVPR/ICCV/NeurIPS/ACL 等顶会模板 |
| 模板填充 | LLM + RAG | 根据用户论文库内容填充模板 |

#### 3.6.2 AI 味校验

**问题**：LLM 生成的学术文本常有明显的"AI 味道"：

- 滥用破折号 `——`、长破折号 `——`
- 高频套话：`"值得注意的是"`、`"综上所述"`、`"首先...其次...最后"`、`"不仅...而且..."`
- 空洞的转折：`"然而"` 每段一次、`"此外"` 滥用
- 过度概括：`"在当今时代"`、`"随着...的发展"`、`"具有重要的理论意义和实践价值"`
- emoji / 非学术符号

**方案：两层防线**

```
Layer 1: Prompt 层面禁止（事前预防）
─────────────────────────────────────
在 Writing Agent 的所有生成 prompt 中追加 SYSTEM 指令：

"""
【写作风格要求】
- 禁止使用中文破折号 "——"，用句号断开或用逗号
- 禁止使用以下套话：
  * "值得注意的是" / "值得一提的是"
  * "综上所述" / "总而言之" / "综上所述"
  * "在当今时代" / "随着...的发展" / "具有重要的理论意义和实践价值"
  * "不仅...而且..." / "既...又..."
- "然而" / "此外" / "因此" 每段最多出现 1 次
- 禁止使用任何 emoji 或非学术符号（如 →、★、✅）
- 直接陈述事实，不要用 "首先...其次...最后" 的排比结构
- 使用学术写作的自然语言风格，如同该领域研究者写的那样
- 参考你知识库中已入库论文的写作风格
"""
```

```
Layer 2: 生成后校验（事后检测）
─────────────────────────────────
Writing Agent 内部增加 ai_flavor_check 节点：

1. 正则检测
   - "——" → 标记（建议替换为句号或逗号）
   - "首先.*其次.*最后" → 标记
   - "值得注意的是" → 标记

2. 频率检测
   - "然而" 出现 > N段 → 标记
   - "此外" 出现 > N段 → 标记
   - 段落开头连续用 "另外/同时/此外" → 标记

3. LLM judge（轻量，仅对标记段落执行）
   Prompt: "检查以下学术文本是否有 AI 生成痕迹。
           如果有，指出具体问题并给出修改建议。
           如果文本自然流畅，返回 passed=true。"

4. 处理策略
   - ≥3 个问题 → 返回给用户标注 ⚠️，建议"一键去 AI 味"
   - 1-2 个问题 → 自动修正后展示（标注 [已自动修正]）
   - 0 个问题 → 直接展示
```

**AI 味黑名单配置**：

```python
AI_FLAVOR_BLACKLIST = {

        "→": "建议替换为 '用于' / '以' / '从而'",
        "★": "禁止使用",

        "❌": "禁止使用",

### 3.7 Tool 共享机制：主 Agent 规划、子 Agent 自主调用

#### 3.7.1 设计原则

| 层级 | 职责 | 具体内容 |
|------|------|----------|
| **Main Agent** | 规划层面 | 决定场景、列出子 Agent、指定 `depends_on` 编排、可选 `tool_hints` |
| **子 Agent** | 执行层面 | 根据场景上下文自主决定具体 tool 调用序列和参数 |
| **Tool** | 原子能力 | 无状态、可被任意子 Agent 注册调用 |

#### 3.7.2 两层 Tool 指定机制

```
Main Agent ScenarioPlanResult.tools[]:
┌────────────────────────────────────────────────────────────┐
│ tool_calls: [                                               │
│   {                                                         │
│     call_id: "lit-1",                                      │
│     kind: "sub_agent",                                     │
│     name: "LiteratureAgent",                                │
│     arguments: { query: "自动驾驶视觉对抗攻击" },           │
│     tool_hints: ["search_across_sources",                   │
│                   "glossary.collect"],  ← 主Agent建议的tool  │
│     depends_on: []                                          │
│   }                                                         │
│ ]                                                           │
└────────────────────────────────────────────────────────────┘
                         │
                         ▼
Literature Agent 内部 tool_selection 节点:
┌────────────────────────────────────────────────────────────┐
│ 收到 tool_hints: ["search_across_sources", "glossary.collect"]│
│                                                            │
│ 子 Agent LLM 判断：                                         │
│   - search_across_sources → 转为具体调用:                    │
│       search(arxiv, query="adversarial attack autonomous")  │
│       search(semantic_scholar, query="physical adversarial")│
│       search(cnki, query="自动驾驶对抗攻击")                 │
│   - glossary.collect → 暂不执行（等 Knowledge Agent 入库后）│
│                                                            │
│ 同时子 Agent 也可自主添加 tool：                              │
│   - download_pdfs（主 Agent 没建议但子 Agent 判断需要）      │
└────────────────────────────────────────────────────────────┘
```

#### 3.7.3 关键规则

1. **主 Agent 指定是 hints，不是强制命令**：`tool_hints` 是建议，子 Agent 可调整、跳过、追加
2. **主 Agent 也可强制指定**：如果 `tool_hints` 中标记 `required: true`，子 Agent 必须执行
3. **Tool 注册去重**：同一 tool 被多个 Agent 注册时，`tool_registry` 只存一份，通过 ACL 控制可见性
4. **Tool 所有权**：Tool 有 `owner_agent` 字段表示开发归属，但不影响调用权限

#### 3.7.4 Tool Registry ACL 设计

```python
class ToolSpec:
    name: str                       # "search_across_sources"
    owner_agent: str                # "LiteratureAgent"
    allowed_agents: list[str] = []  # 空列表 = 所有 Agent 可调用
    
    # 特殊 tool：多 Agent 共享
    shared_tools: list[str] = [
        "glossary_agent.*",          # Glossary Sub-Agent 的所有 tool
        "citation_verify",          # 引用校验（Writing/Research 都可用）
        "translate_en_zh",          # 翻译（多个 Agent 都可用）
    ]
```

### 3.8 多 Agent 协作方案

#### 3.8.1 主 Agent 规划是否足够？

主 Agent 集中规划是当前方案的核心，但它有局限：

| 局限 | 表现 | 解决方案 |
|------|------|----------|
| **无法预判子 Agent 结果** | 主 Agent 在 execute_plan 前就要决定所有 tool_calls，但 Literature Agent 搜到 50 篇和搜到 5 篇论文的后续处理完全不同 | **动态 Re-Plan**：evaluate_completion 节点支持 plan-iterate（当前已支持，最多 3 轮） |
| **子 Agent 间直接协作** | Literature Agent 搜到一篇"方法对比"论文 → Research Agent 应该直接收到信号，无需绕回 Main Agent | **子 Agent 可发起协作请求**（见 §3.8.2） |
| **知识传递效率** | 每个子 Agent 完成后，结果要序列化 → Main Agent → 反序列化传给下一个 Agent | **共享上下文池**（见 §3.8.3） |

**结论**：主 Agent 规划是足够的，但需要补充 **动态 Re-Plan**（已有）和 **子 Agent 间短路径协作**（新增）。

#### 3.8.2 子 Agent 间短路径协作

当前流程：
```
Literature Agent → Main Agent → Knowledge Agent → Main Agent → Writing Agent
       ↑                    ↑                    ↑
    (串行，每次回 Main Agent)
```

优化后：
```
Literature Agent ──→ Main Agent（规划层）
       │                    │
       ├── 论文列表 ────────→ Knowledge Agent（直接传递，不绕 Main Agent）
       │                    │
       └── 关键发现 ────────→ Research Agent（子 Agent 间协作请求）
                              │
                              └──→ Main Agent（通知即可，不阻塞）
```

**实现**：

```python
# 子 Agent 间协作消息
class CollaborationMessage(BaseModel):
    from_agent: str             # "LiteratureAgent"
    to_agent: str               # "KnowledgeAgent"
    intent: str                 # "start_ingest" / "relay_findings" / "request_analysis"
    payload: dict               # 传递的数据
    notify_main: bool = True    # 是否同步通知 Main Agent

# Literature Agent 完成搜索后
async def _on_search_complete(results):
    # 1. 直接触发 Knowledge Agent（短路径）
    await collaboration_bus.send(CollaborationMessage(
        from_agent="LiteratureAgent",
        to_agent="KnowledgeAgent",
        intent="start_ingest",
        payload={"paper_ids": results, "mode": "full"},
    ))

    # 2. 如果发现了关键对比论文，直接通知 Research Agent
    if results.has_comparison_papers():
        await collaboration_bus.send(CollaborationMessage(
            from_agent="LiteratureAgent",
            to_agent="ResearchAgent",
            intent="relay_findings",
            payload={"comparison_papers": results.comparison_papers},
        ))

    # 3. 通知 Main Agent（不阻塞）
    if notify_main:
        await outbox_publish(Message(type="collaboration", payload={...}))
```

**Main Agent 的角色**：
- 接收协作通知（不阻塞流程）
- 如果子 Agent 间的协作结果影响整体计划 → 触发 Re-Plan
- 如果协作正常 → 仅在进度中展示

#### 3.8.3 共享上下文池

多 Agent 协作时，避免每个 Agent 重复查询相同数据：

```python
class SharedContextPool:
    """子 Agent 间共享的上下文缓存"""
    
    def __init__(self, redis_client):
        self.redis = redis_client
        self.pool_key = f"context:{task_id}"
    
    async def put(self, key: str, value: dict, ttl: int = 3600):
        """写入共享上下文（如 Literature Agent 的搜索结果）"""
        await self.redis.hset(self.pool_key, key, json.dumps(value))
        await self.redis.expire(self.pool_key, ttl)
    
    async def get(self, key: str) -> dict:
        """读取共享上下文（如 Knowledge Agent 读取搜索结果）"""
        data = await self.redis.hget(self.pool_key, key)
        return json.loads(data) if data else None
```

**典型流程**：

```
task_id = "task-123"

Literature Agent:
  context_pool.put("search_results", {papers: [...]})
  context_pool.put("search_query", {query: "自动驾驶对抗攻击"})

Knowledge Agent (稍后启动):
  papers = context_pool.get("search_results")  # 直接拿到 Literature 的结果
  # 直接入库，不需要再查一次

Writing Agent (稍后启动):
  query = context_pool.get("search_query")     # 知道用户在调研什么
  # 生成更有针对性的综述
```

| 场景 | 触发方 | 协作对象 | 传递内容 |
|------|--------|----------|----------|
| 文献调研→入库 | Literature Agent | Knowledge Agent | 论文 ID 列表 + 搜索 query |
| 入库→术语收集 | Knowledge Agent | Glossary Sub-Agent | 新入库论文 ID 列表 |

| 聚类→综述 | Research Agent | Writing Agent | 聚类结果（方向标签 + 代表论文） |

| RAG→校验 | Knowledge Agent | Knowledge Agent（内部） | 生成的引用列表 + chunk 映射 |

## 4. LLM 输出引用标记与内外文献验证方案

### 4.0 校验范围与职责

#### 4.0.1 什么需要校验

**LLM 输出并非全部需要引用校验**。校验范围取决于 LLM 的回答来源：

```
LLM 输出场景分类：

场景 A: 纯 LLM 通用知识回答（无需校验）
  用户："Python 的 GIL 是什么？"
  LLM 回答（基于训练知识）→ 不校验，允许自由发挥
  标记：无 [local:xxx] 或 [ext:xxx]

场景 B: RAG 检索后生成的回答（需要校验）
  用户："我知识库中对抗攻击论文的结论是什么？"
  Knowledge Agent RAG 检索 → LLM 综合生成 → 必须校验每一条引用
  标记：必须有 [local:pap-xxx] ✓/⚠️/❌

场景 C: 文献搜索后综合分析（需要校验）
  用户："调研 2024 年自动驾驶对抗攻击方法"
  Literature Agent 搜索 → LLM 综述生成 → 必须校验新搜索到的文献
  标记：新搜索到的论文 [ext:doi]，知识库已有论文 [local:pap-xxx]

场景 D: 纯 Agent 综合判断（部分校验）
  用户："我的研究方向有哪些空白可以探索？"
  Research Agent 聚类 → LLM 分析 → 引用的具体论文需校验，综合判断不校验
  标记：具体论文引用 [local:pap-xxx]，综合判断 [Agent 综合]
```

**核心原则**：给 LLM 保留发挥空间，不强制每句话都引文献。但明确在 prompt 中规定：**如果你的回答使用了知识库中的论文信息、或者搜索到了新论文，必须标注来源**。

```
Prompt 中的关键指令：

"""
关于引用标注：
- 如果你的回答基于知识库检索到的论文，必须用 [local:pap-xxx] 标注
- 如果你引用了搜索到的新论文，必须用 [ext:doi] 标注
- 如果你的回答是综合多篇论文的独立判断，标注 [Agent 综合]
- 如果来源不明确（不确定来自哪篇论文），标注 [来源待核查]
- 不要为了引用而引用——没有来源支撑的观点不需要编造引用
- 保持学术表达的流畅性，不要把引用标注变成"拼凑答案"
"""
```

#### 4.0.2 谁来校验：Knowledge Agent

校验职责归属 **Knowledge Agent**（而非 Writing Agent），理由：

| 维度 | Knowledge Agent | Writing Agent |
|------|:---:|:---:|
| 校验所需的论文全文 | 有（pgvector chunk） | 无 |
| 校验所需的引用上下文 | 有（citations 表） | 无 |
| 外部验证能力 | 有（external_validator） | 无 |
| 与 RAG 检索同属一个 Agent | 是 | 否 |

**流程**：

```
Main Agent execute_plan → 子 Agent 执行完毕 →
    │
    ├── Literature Agent 完成（搜索到了新论文）
    │   └── Main Agent 自动追加 KnowledgeAgent.verify_citations
    │       → 校验搜索结果中 LLM 引用了哪些外部文献
    │
    ├── Knowledge Agent 完成（RAG 回答）
    │   └── 内部 output_verify 节点自动校验所有 [local:xxx] 引用
    │
    ├── Writing Agent 完成（综述/related work）
    │   └── Main Agent 自动追加 KnowledgeAgent.verify_citations
    │       → 校验综述中引用的文献是否真实存在
    │
    └── 纯聊天 / 简单问答
        → 不触发校验
```

**避免硬答**：LLM 面临"数据库中没有相关信息，但被要求回答"时，策略如下：

```
if 检索结果数量为 0:
    LLM 回答: "知识库中暂未找到相关信息。你可以[发起文献调研]或[上传相关论文]。"
    不生成引用标记
    
if 检索结果质量低（相似度 < 阈值）:
    LLM 回答: "检索到部分信息但相关度较低：[Agent 综合] 基于有限信息，..."
    标记为 [Agent 综合] 而非 [local:xxx]
    不做强制校验
```

### 4.1 设计目标

LLM 输出的每一句话，如果来源于文献，必须标记：
1. **来源类型**：库内文献（local） vs 外部文献（external）
2. **文献标识**：paper_id（库内）或 DOI（外部）
3. **验证状态**：✓ 已验证 / ⚠️ 待核查 / ❌ 验证失败
4. **引用粒度**：尽可能定位到章节（如 `#sec-3.2`）

### 4.2 输出格式规范

#### 4.2.1 综述 / RAG 回答格式

```markdown
## 研究背景

近年来，自动驾驶视觉对抗攻击受到广泛关注 [local:pap-001] ✓。
其中，基于物理世界的攻击方法在真实场景中展现出更高的威胁性 [local:pap-002#sec-3] ✓。

> "Physical adversarial attacks pose a significant threat to real-world autonomous driving systems." [local:pap-002#sec-1] ✓

不同工作对攻击面的定义存在分歧 [Agent 综合]，主要分为数字域攻击和物理域攻击两大类。
```

#### 4.2.2 标记规范

| 标记格式 | 含义 | 示例 |
|----------|------|------|
| `[local:{paper_id}]` | 库内已验证文献 | `[local:pap-001]` |
| `[local:{paper_id}#{section}]` | 库内文献 + 章节定位 | `[local:pap-002#sec-3.2]` |
| `[ext:{doi}]` | 外部文献（通过 Crossref 验证） | `[ext:10.1145/3293318]` |
| `[ext:arxiv:{arxiv_id}]` | 外部 arXiv 文献 | `[ext:arxiv:2301.12345]` |
| `[Agent 综合]` | LLM 基于多篇文献的综合判断 | 不加具体 paper_id |
| `[来源待核查]` | 无法验证来源的声明 | 红色警示 |

#### 4.2.3 验证状态图标

| 图标 | 含义 | 条件 |
|------|------|------|
| ✓ | 已验证通过 | 库内验证 OR 外部验证任一通过 |
| ⚠️ | 待核查 | 库内+外部均未通过，但外部验证不可达（非论文不存在） |
| ❌ | 验证失败 | 库内+外部均确认该引用无法验证，已通知用户 / 触发重生成 |

### 4.3 内外文献双通道验证流程（任一通过对即可放行）

```
LLM 生成输出（含 [local:xxx] / [ext:xxx] 标记）
              │
              ▼
    ┌─────────────────────┐
    │   Citation 解析器    │  提取所有 [local:xxx] 和 [ext:xxx]
    └─────────┬───────────┘
              │
              ▼
    ┌─────────────────────────────────────────┐
    │  对每条引用，两条通道并行验证             │
    │                                         │
    │  ┌─────────────────┐  ┌───────────────┐ │
    │  │ 库内验证 (L2)    │  │ 外部验证 (L3) │ │
    │  │                 │  │               │ │
    │  │ 1. paper_id查DB │  │ 1. DOI→Crossref│ │
    │  │ 2. chunk匹配    │  │ 2. arxiv→arXiv│ │
    │  │ 3. claim校验   │  │ 3. title→S2   │ │
    │  └────────┬────────┘  └───────┬───────┘ │
    │           │                   │         │
    │           └───── OR ──────────┘         │
    │                     │                   │
    │           ┌─────────┴─────────┐         │
    │           │ 任一通过对 → 放行  │         │
    │           └───────────────────┘         │
    └────────────────────┬────────────────────┘
                         │
                         ▼
            每条引用的验证结果
                         │
            ┌────────────┼────────────┐
            ▼            ▼            ▼
      库内通过      外部通过      两个都失败
      → ✓ 放行      → ✓ 放行      → ❌
                                   │
                          ┌────────┴────────┐
                          ▼                 ▼
                   通知用户处理       LLM 自动重生成
                   （标注 ⚠️）       （带失败原因） 
```

**核心原则**：

- **任一通过对 → 放行**：库内验证 OR 外部验证只要一个确认该论文真实存在，引用即可放行
- **两个都失败 → 分情况处理**：
  - 用户可见输出中标注 `[来源待核查]` ⚠️，同时通知用户："以下引用未能验证，请核实"
  - 同时触发 LLM 自动重生成：将失败的引用列表 + 失败原因喂回 LLM，让 LLM 删除不可靠引用或替换为可验证的替代文献
  - 重生成后再次走验证，最多重试 2 次
- **外部验证不可达不算失败**：如果 Crossref/arXiv API 超时或不可用，标记为 `[外部源不可达]` 而非 `[来源待核查]`，仅降级显示但不阻塞输出

### 4.4 库内验证通道（L2 — CitationVerifier 三步）

当前 `verifier.py` 已实现三步校验（DOI 查证 → 标题匹配 → URL 可访问性）。重构后扩展为：

```python
class LocalCitationVerifier:
    """库内引用验证器"""

    async def verify(self, citation: LocalCitation) -> CitationVerdict:
        # Step 1: paper_id 查 AgentDB —— 论文是否存在
        paper = await db.get_paper(citation.paper_id)
        if not paper:
            return CitationVerdict(action="delete", reason="paper_not_found")

        # Step 2: chunk 匹配 —— 论文是否包含被引内容
        chunks = await self._search_chunks(
            query=citation.claim_text,
            filter_paper_id=citation.paper_id,
            top_k=3
        )
        if not chunks or chunks[0].score < 0.5:
            return CitationVerdict(action="flag", reason="claim_not_supported")

        # Step 3: claim 校验 —— 用 LLM judge 判断论文是否真的说了 LLM 声称的内容
        supported = await self._llm_judge_claim(citation.claim_text, chunks)
        if not supported:
            return CitationVerdict(action="flag", reason="claim_mismatch")

        return CitationVerdict(action="keep", paper_id=paper.id, section=chunks[0].heading)
```

### 4.5 外部验证通道（L3 — ExternalValidator）

当前 `external_validator.py` 已有骨架。重构后：

```python
class ExternalCitationVerifier:
    """外部文献验证器 —— 引用不在库内时走此通道"""

    async def verify(self, citation: ExternalCitation) -> ExternalVerdict:
        # Step 1: 优先 DOI 直查（确定性最高）
        if citation.doi:
            result = await self._crossref_lookup(citation.doi)
            if result.exists:
                return ExternalVerdict(action="keep", source="crossref", doi=result.doi)

        # Step 2: arXiv ID 查 arXiv API
        if citation.arxiv_id:
            result = await self._arxiv_lookup(citation.arxiv_id)
            if result.exists:
                return ExternalVerdict(action="keep", source="arxiv")

        # Step 3: 标题 + 作者模糊匹配 Semantic Scholar
        if citation.title and citation.authors:
            result = await self._s2_search(citation.title, citation.authors)
            if result.exists and result.confidence > 0.8:
                return ExternalVerdict(action="keep", source="semantic_scholar")

        # 全部未命中
        return ExternalVerdict(action="delete", reason="external_not_found")
```

### 4.6 内外文献验证策略差异

| 维度 | 库内文献（local） | 外部文献（external） |
|------|:---:|:---:|
| **验证深度** | 三步（存在性 + chunk 匹配 + claim 校验） | 两步（存在性 + 元数据匹配） |
| **能否做 claim 校验** | 能（有全文 chunk） | 否（无全文，仅元数据） |
| **验证失败处理** | flag + 原因 | delete（不展示不可靠外部引用） |
| **输出图标** | ✓ / ⚠️ | ✓ / ❌（不用 ⚠️） |
| **缓存策略** | 无需缓存（本地查） | 30 天缓存（external_validations 表） |

#### 4.9.1 竞品对比

### 5.2 为什么是子 Agent 而非 Tool

| 维度 | Tool | Sub-Agent（推荐） |
|------|------|:---:|
| **职责范围** | 单次调用、无状态 | 持续维护词表，有长期记忆 |
| **学习能力** | 每次从零开始 | frequency 累积、用户反馈学习 |
| **集成复杂度** | 每个调用方自己拼参数 | 统一入口，内部决策策略 |
| **术语质量进化** | 每次独立的 TF-IDF+LLM | 历史词表作为先验，新术语对比已有词表 |
| **多步骤编排** | 调用方需知道 4 步流程 | 子 Agent 内部编排，调用方只需给 paper_ids |

**结论**：术语词表有"越用越好"的长期记忆需求（frequency 累积、用户确认反馈），适合封装为子 Agent。

### 5.3 生成流程（Literature Agent 触发）

#### 5.3.1 输入：MD 而非 PDF

术语提取的输入必须是 **PDF→MD 转换后的 Markdown 文件**，因为：

- PDF 提取的原始文本可能格式混乱（双栏排版、换行符错位）
- MD 中保留了公式（`$...$` / `$$...$$`）、表格、标题层级
- MD 可以直接做正则分词，无需 PDF 解析的额外复杂度

#### 5.3.2 完整流水线

```
Literature Agent 完成 search → download → convert(PDF→MD)
    │
    ▼
┌──────────────────────────────────────────────────────────────┐
│ Step 1: 分词 + 跨文章词频统计                                  │
│                                                              │
│ 对每篇论文的 MD 文件：                                        │
│   1. 按段落切分（保留 section 层级）                           │
│   2. 每个段落：                                               │
│      a. 去除 Markdown 标记（## / ** / [] / () / ![]() ）     │
│      b. 保留 LaTeX 公式内容（$...$ / $$...$$ 内的文本）       │
│      c. 英文按空格 + 标点分词                                 │
│      d. 提取 n-gram（1,2,3-gram）                            │
│   3. 全局聚合：统计每篇文章的词频 + 跨文章的文档频率（DF）      │
│                                                              │
│ 技术选型：scikit-learn CountVectorizer + TfidfVectorizer     │
│ 关键参数：                                                    │
│   - ngram_range=(1, 3)            # 1-3 gram                │
│   - max_df=0.8                    # 出现在 >80% 文档中 → 停用 │
│   - min_df=2                      # 至少出现在 2 篇文档中      │
│   - stop_words='english'          # 去掉通用停用词            │
│   - max_features=200              # 候选池 200 个             │
│                                                              │
│ 自定义停用词（学术论文特有）：                                 │
│   paper, propose, method, result, experiment, show,         │
│   demonstrate, figure, table, et al, approach, based,      │
│   using, used, also, however, therefore, furthermore        │
│                                                              │
│ 产出：200 个候选术语（含频率 + 跨文档数）                      │
└──────────────────────┬───────────────────────────────────────┘
                       ▼
┌──────────────────────────────────────────────────────────────┐
│ Step 2: 跨文章高频词加权                                       │
│                                                              │
│ 重要词汇的两个维度：                                           │
│   TF (Term Frequency)    = 单篇文章内出现多少次               │
│   DF (Document Frequency) = 多少篇文章中出现这个词             │
│                                                              │
│ 加权公式（偏好多篇文章共同出现）：                              │
│   score = α × TF_normalized + (1-α) × DF / N                │
│   其中 α = 0.3（DF 权重更大）                                 │
│                                                              │
│ 例如：                                                        │
│   "adversarial":   TF高 + DF=15/20 → score=0.82  ★高优先级  │
│   "backpropagation": TF中 + DF=20/20 → score=0.75 ★         │
│   "tachyon":       TF高 + DF=1/20  → score=0.35  低优先级   │
│                                                              │
│ 过滤：score < 0.3 → 丢弃（可能是某篇论文的特有词）            │
│ 排序：按 score 降序取 Top-100                                 │
└──────────────────────┬───────────────────────────────────────┘
                       ▼
┌──────────────────────────────────────────────────────────────┐
│ Step 3: 短语合并（n-gram 去重）                                │
│                                                              │
│ "adversarial" + "attack" → "adversarial attack" (bigram)     │
│ "autonomous" + "driving" → "autonomous driving"              │
│                                                              │
│ 规则：如果 bigram 的 DF ≥ unigram 中较低者 × 0.6，保留 bigram │
│ 同时丢弃被合并的 unigram（除非该 unigram 有其他组合）          │
└──────────────────────┬───────────────────────────────────────┘
                       ▼
┌──────────────────────────────────────────────────────────────┐
│ Step 4: LLM 重排序 + 决定是否入库                               │
│                                                              │
│ 将 Top-100 候选术语 + 上下文（所属论文领域、频次）发给 LLM：    │
│                                                              │
│ Prompt:                                                      │
│ "以下是来自{domain}方向的 100 个高频候选术语。                 │
│  请判断每个术语是否应该收入术语词表。                          │
│  标准：                                                      │
│    - 专业性强：是该领域的专有术语（如 adversarial attack）     │
│    - 区分度高：不是通用学术词（如 experiment, result）        │
│    - 翻译价值：中英对照有助于检索/写作/翻译                   │
│  对每个术语输出：{en_term, zh_term, should_keep: bool,       │
│                  confidence: 0-1, reason}                    │
│  只保留 should_keep=true 的术语"                              │
│                                                              │
│ 产出：50-80 个审核通过的术语（带中文翻译 + 置信度）           │
└──────────────────────┬───────────────────────────────────────┘
                       ▼
┌──────────────────────────────────────────────────────────────┐
│ Step 5: 去重 + 入库 + 更新记忆                                 │
│                                                              │
│ - 与 glossary_terms 表已有术语比对（en_term + zh_term 联合）  │
│ - 新术语 → INSERT (frequency=1, confidence=LLM给的分数)       │
│ - 已有术语 → UPDATE frequency+=1,                            │
│               如果新翻译与旧翻译不同 → 追加到 variants         │
│ - 写入 pgvector glossary_embeddings 表（供模糊语义匹配）          │
│ - 记录 source_paper_ids（可追溯术语来源）                     │
└──────────────────────────────────────────────────────────────┘
```

#### 5.3.4 LaTeX 公式处理策略

```

| 10 | **研究人员切换方向** | Main Agent 调度 | `glossary.list_by_domain(new_domain)` | 用户切换研究方向后，加载新方向术语作为先验 |

#### 5.8.4 召回率评估方法

```python
class RecallEvaluator:
    """跨语言检索召回率评估器"""

    def evaluate(self, test_set: list[TestQuery]) -> RecallReport:
        """
        test_set: 人工标注的测试集
        每个 TestQuery 包含：
          - chinese_query: str
          - expected_paper_ids: list[str]  # 人工确认相关的论文
        """
        results = []
        for query in test_set:
            # 方案 A: 纯中文 embedding
            baseline = self._search_baseline(query.chinese_query)
            # 方案 B: 术语库扩展
            expanded = self._search_with_glossary(query.chinese_query)
            # 方案 C: 多 query 并行
            multi = self._search_multi_query(query.chinese_query)

            results.append({
                "query": query.chinese_query,
                "baseline_recall@10": self._recall_at_k(baseline, query.expected_paper_ids, k=10),
                "expanded_recall@10": self._recall_at_k(expanded, query.expected_paper_ids, k=10),
                "multi_recall@10": self._recall_at_k(multi, query.expected_paper_ids, k=10),
            })

        return RecallReport(
            avg_baseline_recall=mean(r["baseline_recall@10"] for r in results),
            avg_expanded_recall=mean(r["expanded_recall@10"] for r in results),
            avg_multi_recall=mean(r["multi_recall@10"] for r in results),
            per_query=results,
        )
```

### 5.9 术语词表编排：何时触发、如何通知用户

#### 5.9.1 触发时机

术语收集在 **Knowledge Agent 完成论文入库后自动触发**，无需用户手动操作：

```
Literature Agent 完成文献调研
    │
    ▼
Knowledge Agent 完成 chunk + embed + 入库
    │
    ├──→ 同步返回：Main Agent 执行流程中的下一个子 Agent
    │
    └──→ 异步触发：Celery 任务 celery_glossary_collect.delay(paper_ids, user_id)
              │
              │  (后台静默执行，不阻塞主流程)
              ▼
         Glossary Sub-Agent 执行 collect 流水线
              │
              │  (5-30 秒后完成，取决于论文数量)
              ▼
         Main Agent 收到完成回调 → 推送一条文本消息到 Chat
```

#### 5.9.2 用户通知

采用极简通知：**一条普通文本消息**，与 Chat 中的其他消息风格一致：

```
Chat 中展示：

  [10:32] ✅ 文献调研完成！已入库 22 篇论文。

  [10:35] 📝 术语词表已更新：从本次调研中提取 35 个专业术语，可用于后续检索和写作。

```

#### 5.9.3 实现

```python
# Knowledge Agent embed 节点完成后
async def _after_embed(state: KnowledgeAgentState):

    # 推送到 Main Agent 的 outbox

# Main Agent 收到 outbox 消息后
# 通过 WebSocket 推送一条普通文本消息到前端 Chat

```

#### 5.9.4 关键决策

| 决策 | 选择 | 理由 |
|------|------|------|
| 同步 vs 异步 | 异步（Celery） | 术语收集 5-30 秒，不应阻塞主流程 |
| 通知形式 | 普通文本消息 | 与 Chat 风格一致，不引入特殊组件 |
| 通知时机 | collect 完成后 | 用户看到的是结果，不是中途进度 |
| 通知内容 | 仅一句话 + 术语数量 | 避免冗余，用户想深入了解可去术语管理页面 |
| 是否支持跳过 | 始终执行 | 术语收集对后续检索至关重要，不应跳过 |

#### 5.9.5 术语词表维护机制

- **入库触发**：每次 Knowledge Agent 入库完成 → Celery 异步触发术语收集
- **人工审核**：Web 前端提供术语管理页面（`/glossary`），用户可确认/修改/删除术语
- **去重**：same en_term + zh_term 自动合并，variants 并集，frequency 求和
- **置信度更新**：同一术语被 ≥3 篇论文使用时 confidence → 1.0

## 6. 去重策略

### 6.1 当前方案（问题）

- `ingest_graph.py` 中通过 embedding 相似度去重
- 问题：embedding 相似度无法区分"同一篇论文的不同版本"和"不同论文但内容相似"

### 6.2 重构方案：两级去重

```
论文入库
    │
    ▼
┌─────────────────────┐
│ Level 1: 精确去重    │  基于 title + journal/conference 匹配
│                     │
│ 规则：               │
│ 1. title 归一化      │   小写、去标点、去多余空格
│    （编辑距离 < 3）   │
│ 2. journal/conference│   同一会议不同年份 → 不同论文
│    名称匹配           │   同一会议同一年份 → 可能重复
│ 3. year 匹配         │   同年 + 同标题 → 重复
│                     │
│ 保留策略：           │
│ - CVPR/ICCV/ECCV    │   CCF-A 会议优先
│   > NeurIPS/ICML    │
│   > 其他会议/期刊    │
│   > arXiv preprint  │   非权威来源最低优先级
└─────────┬───────────┘
          │ 未命中
          ▼
┌─────────────────────┐
│ Level 2: 语义去重    │  基于 embedding 的兜底策略
│                     │
│ - title embedding    │   相似度 > 0.95
│ - abstract embedding │   交叉验证
│ - 保留权威来源       │   同 L1 规则
└─────────────────────┘
```

### 6.3 保留优先级规则

```python
SOURCE_PRIORITY = {
    # CCF-A 会议
    "cvpr":      100,
    "iccv":      100,
    "eccv":      100,
    "neurips":   95,
    "icml":      95,
    "acl":       95,
    "aaai":      90,
    "ijcai":     90,
    # CCF-A 期刊
    "tpami":     100,
    "ijcv":      100,
    "tip":       90,
    # CCF-B 会议/期刊
    "emnlp":     80,
    "eccv_workshop": 70,
    # arXiv
    "arxiv":     50,
    # 其他
    "unknown":   30,
}

def resolve_duplicate(paper_a: Paper, paper_b: Paper) -> Paper:
    """当两篇论文判定为重复时，保留权威来源"""
    priority_a = SOURCE_PRIORITY.get(paper_a.source, 30)
    priority_b = SOURCE_PRIORITY.get(paper_b.source, 30)

    if priority_a >= priority_b:
        kept, discarded = paper_a, paper_b
    else:
        kept, discarded = paper_b, paper_a

    # 记录去重关系
    kept.duplicate_of = None
    kept.alternate_versions = (kept.alternate_versions or []) + [discarded.id]
    discarded.duplicate_of = kept.id

    return kept
```

## 7. ASR 准确率提升方案

### 7.1 问题分析

视频/音频转文本（ASR）在科研场景的核心痛点是**专业术语识别率低**。通用 ASR 模型（如 Whisper）对"adversarial attack"、"occupancy prediction"、"backpropagation"等术语容易识别错误。

### 7.2 解决方案：术语增强 ASR

```
┌─────────────────────────────────────────────────────────────┐
│               术语增强 ASR 流程                              │
│                                                             │
│  音频文件                                                   │
│     │                                                       │
│     ├──→ [ASR 引擎] (Whisper / 火山引擎 ASR)                 │
│     │       │                                               │
│     │       ▼ 原始转写文本                                    │
│     │                                                       │
│     ├──→ [术语后处理]                                        │
│     │       │                                               │
│     │       ├─ 从 glossary 加载用户当前研究方向高频术语        │
│     │       ├─ 对转写文本做模糊匹配                           │
│     │       ├─ 发现疑似术语但识别错误的位置                   │
│     │       └─ 用正确术语替换                                │
│     │                                                       │
│     ▼                                                       │
│  校正后转写文本 → 摘要 → 入库                                │
└─────────────────────────────────────────────────────────────┘
```

### 7.3 具体实现

```python
class ASRTermEnhancer:
    """基于术语词表的 ASR 后处理校正器"""

    def __init__(self, glossary_terms: list[GlossaryTerm]):
        # 构建术语 → 发音模糊匹配索引
        self.term_index = self._build_phonetic_index(glossary_terms)

    def correct(self, transcript: str) -> str:
        """对 ASR 输出做术语校正"""
        words = transcript.split()
        corrected = []

        for i, word in enumerate(words):
            # 检查当前词是否接近某个术语
            match = self._fuzzy_match_term(word, window=3)
            if match and match.confidence > 0.7:
                corrected.append(match.correct_term)
            else:
                corrected.append(word)

        return " ".join(corrected)

    def _fuzzy_match_term(self, word: str, window: int = 3) -> Optional[TermMatch]:
        """模糊匹配：编辑距离 + 发音相似度"""
        candidates = []
        for term in self.term_index:
            # 编辑距离
            edit_dist = levenshtein(word.lower(), term.en_term.lower())
            # 发音编码 (Metaphone / Double Metaphone)
            phonetic_sim = self._phonetic_similarity(word, term.en_term)

            if edit_dist <= 2 or phonetic_sim > 0.8:
                candidates.append(TermMatch(
                    term=term,
                    confidence=1.0 - (edit_dist / max(len(word), len(term.en_term)))
                ))

        return max(candidates, key=lambda c: c.confidence) if candidates else None
```

### 7.4 术语词表对 ASR 的反哺

文献调研沉淀的高频术语自动成为 ASR 校正词典：

1. **入库时收集**：ingest_graph 完成后自动扫描 title/abstract/keywords，提取 TOP-N 术语
2. **按研究方向分组**：用户的 clustering 结果关联术语
3. **ASR 时按场景加载**：如果用户在看 CV 相关视频，只加载 CV 方向术语（减少误匹配）
4. **反馈闭环**：ASR 校正后如果用户手动修正了某处，记录为训练数据

### 7.5 评估方法

| 指标 | 说明 | 目标 |
|------|------|------|
| 术语识别准确率 | 转写文本中专业术语的正确率 | 基线 → +15% |
| WER (词错误率) | 整体转写质量 | 基线 → -10% |
| 校正覆盖率 | 被术语词表覆盖的专业词汇比例 | ≥ 80% |

## 8. 主 Agent 状态管理与并行子 Agent 调度

### 8.1 当前方案

```
Main Agent → execute_plan → 按 depends_on 拓扑排序逐批执行 tools[]
    │
    └── Celery sub_agent_task.delay() 派发子 Agent
        └── Redis Pub/Sub agent:reports:{task_id} 收状态

问题：
  1. execute_plan 是串行拓扑排序 → 同一批内的 tool 才并行
  2. 子 Agent 完成后 reporter → Redis → Main Agent 收 lifecycle，链路过长
```

### 8.2 LangGraph Store 为什么不适合

| 原因 | 说明 |
|------|------|
| Store 是长期记忆 | 设计用于 preferences/profile/episodes/topics 等跨 session 持久化知识 |
| 子 Agent 状态是运行时的 | task_id/进度/stage 是瞬时状态，生命周期 ≤ 30 分钟，不应写入长期记忆 |
| 语义污染 | 把"论文入库进度 12/22"和"用户偏好 tone=concise"放在同一个 Store 里概念混乱 |

### 8.3 推荐方案：Redis + reporter 增强（当前已有，增强即可）

当前已有机制足够，只需增强：

```
Main Agent execute_plan
    │
    ├── 调 celery_tasks.sub_agent_task.delay() 派发子 Agent
    │
    ├── 轮询 Redis Pub/Sub channel: agent:reports:{task_id}
    │   ├── type=progress → 更新 state.tool_results
    │   ├── type=lifecycle + agent_done → 标记该 task 完成
    │   └── type=lifecycle + agent_failed → 标记失败
    │
    └── 超时：sub_agent 默认 30 分钟，超时 → 标记 failed + 通知用户
```

**增强点**：

| 当前 | 增强后 |
|------|--------|
| reporter 只推送 lifecycle（done/failed） | 增加 progress 事件：`{stage, progress, papers[]}` |
| Main Agent 等待所有 task 完成后才 evaluate | 支持中间进度可见——每 batch 完成后推 `sub_agent/progress` 到 outbox |
| task_id 无用户关联 | 增加 `user_id` 字段，多用户场景下 Pub/Sub channel 按 user_id 隔离 |

### 8.4 新增：并行子 Agent 调度

现有 `execute_plan` 已支持拓扑分批并行（`depends_on`），但一次场景规划 (scenario_plan) 只产生一个执行计划。某些场景需要 **Main Agent 同时启动多个子 Agent 并行工作**。

#### 8.4.3 execute_plan 并行调度算法

```python
async def _node_execute_plan(state: MainAgentState) -> dict:
    plan = state["plan"]
    all_results = {}

    # 1. 先处理传统 tools[]（向后兼容）
    if plan.tools:
        batch_results = await _execute_parallel_group(
            ParallelGroup(group_id="main", tools=plan.tools)
        )
        all_results.update(batch_results)

    # 2. 处理 parallel_groups
    groups = {g.group_id: g for g in plan.parallel_groups}
    completed_groups = set()

    while len(completed_groups) < len(groups):
        # 找出所有依赖已满足的 group
        ready_groups = [
            g for g in groups.values()
            if g.group_id not in completed_groups
            and all(dep in all_results or dep in completed_groups
                   for dep in g.depends_on)
        ]

        if not ready_groups:
            logger.error("Deadlock in parallel groups")
            break

        # ★ 并行执行所有 ready group（每个 group 内的 tools 也并行）
        tasks = [_execute_parallel_group(g) for g in ready_groups]
        batch_results = await asyncio.gather(*tasks, return_exceptions=True)

        for group, results in zip(ready_groups, batch_results):
            all_results.update(results or {})
            completed_groups.add(group.group_id)

            # 推送进度到用户
            await outbox_publish(Message(
                type="sub_agent/parallel_progress",
                payload={
                    "completed_groups": len(completed_groups),
                    "total_groups": len(groups),
                    "results_summary": _summarize(results),
                }
            ))

    return {"tool_results": all_results}

async def _execute_parallel_group(group: ParallelGroup) -> dict[str, Any]:
    """执行一组 tool（组内并行）"""
    results = {}
    # 按 depends_on 分批（组内 tool 之间也可能有依赖）
    batches = _topo_sort_by_depends(group.tools)
    for batch in batches:
        batch_tasks = [_dispatch_one(tool, state) for tool in batch]
        batch_results = await asyncio.gather(*batch_tasks, return_exceptions=True)
        for tool, result in zip(batch, batch_results):
            results[tool.call_id] = result
    return results
```

### 8.5 子 Agent 崩溃恢复

| 场景 | 处理 |
|------|------|
| 子 Agent Celery task 崩溃 | task 状态变为 FAILURE → Main Agent 收 lifecycle=agent_failed → 判定该 task 失败 → 不阻塞其他并行 task |

## 12. 评估体系

### 12.1 量化指标

#### 12.1.1 检索质量

| 指标 | 定义 | 测量方法 | 目标值 |
|------|------|----------|:---:|
| **Recall@10** | Top-10 结果中相关论文占比 | 人工标注测试集 | ≥ 0.75 |

## 附录 A：竞品对比分析报告

> 完整报告见竞品调研子 Agent 输出，此处摘要核心结论。

### 竞品矩阵

| 维度 | Elicit | Consensus | Scite | ReadPaper | Semantic Scholar | Paper Agent |
|------|:---:|:---:|:---:|:---:|:---:|:---:|
| 端到端自动化 | 部分 | 部分 | 无 | 无 | 无 | **有** |
| 知识沉淀 | 无 | 无 | 无 | 有（笔记） | 无 | **有（知识库+记忆）** |
| 引用验证深度 | 存在性 | 存在性 | **存在性+上下文** | 弱 | 存在性 | **存在性+声明一致性+外部对账** |
| 中文支持 | 无 | 无 | 无 | **全文翻译** | 无 | **中文界面+术语库+跨语言检索** |
| 私有化部署 | 无 | 无 | 无 | 无 | 无 | **有** |
| 定价 | $12-49/月 | $8.99/月 | $20/月 | ~200 元/年 | 免费 | **自部署** |

### 核心差异化

1. **端到端自动化** — 唯一做到"搜→下→转→索→提取→综述"全自动
2. **引用验证** — 唯一做"生成后验证"而非"不生成引用"来规避幻觉
3. **跨语言** — 中英术语库 + 跨语言检索是蓝海
4. **私有化部署** — 面向实验室场景的独特优势
